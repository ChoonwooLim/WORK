v2.2 April 13, 2026

🎬 AI Video Generation (Wan 2.2) + Dedicated GPU Server + Ops Cleanup

  • Feature
    Wan 2.2 AI Video Generation — one-click inside the dashboard

    Integrated Alibaba's Wan 2.2 (the current official open-source SOTA) in three variants: ti2v-5b (5B, fast, ~36s/clip) · t2v-14b (A14B MoE, cinematic) · i2v-14b (A14B MoE, image-to-video). Note: "Wan 2.5" remains Alibaba-API-only as of April 2026 with no open-source weights; we'll add it the moment it's released. A new AI Studio → 🎬 AI Video Generation sidebar section exposes prompt input, model selector, frame/step/resolution controls, a live status chip (GPU / VRAM / loaded model), the player, and an activity log. Orbitron proxies every call via /api/wan/{status,t2v,i2v,video/:filename,healthz} so the LAN IP is never exposed and only authenticated sessions can reach the service. Multipart for image uploads (multer, 20MB cap).

  • Infrastructure
    Dedicated GPU Server (twinverse-ai)

    Split the production Orbitron host from a dedicated AI-inference box: Threadripper 3970X (32C/64T) + RTX 3090 24GB + 128GB DDR4 + dual 10GbE Intel X550, Ubuntu 24.04 LTS. Heavy workloads (31B LLMs, A14B MoE video) moved off the dashboard host, so restarting Orbitron no longer interrupts renders. LAN-only access via UFW (192.168.219.0/24 allowlist on 22/11434/8200/8100), SSH key-only (password auth disabled), systemd linger enabled for boot-time auto-start, nvidia-persistenced + 350W power-limit systemd unit, CUDA Toolkit 12.0 installed. Orbitron routes Gemma 4 chat to the remote GPU via OLLAMA_HOST=http://192.168.219.117:11434: throughput ~46 → ~133 tok/s (2.9×), boot warmup 43.5 → 5.2 s (8.4×).

  • Engineering
    Dynamic GPU VRAM Sharing — triple guardrail

    (1) OLLAMA_KEEP_ALIVE=30s so Gemma only holds VRAM while actively chatting — idle for 30 s and it auto-unloads. (2) The Wan service's _free_ollama_vram() hook posts keep_alive:0 to Ollama right before loading an A14B pipeline, forcing an immediate unload instead of waiting for the timer. (3) enable_sequential_cpu_offload() streams the 28GB-bfloat16 A14B MoE layer-by-layer into 24GB VRAM, and PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True cuts fragmentation.

  • Ops Fix
    Trust proxy, log hygiene, npm audit, Docker prune

    app.set('trust proxy', 1) replaces the permissive true, killing the ERR_ERL_PERMISSIVE_TRUST_PROXY stream and blocking X-Forwarded-For spoofing. morgan now skips /api/projects*/stats dashboard polling, shrinking PM2 out.log from 158MB / 2.5M lines to 47 lines. pm2-logrotate installed (10MB / 7d / compress). A one-shot Gemma 4 warmup request on boot removes the ~30 s first-call cold start. npm audit fix resolved 6 advisories (5 high + 1 moderate: basic-ftp, brace-expansion, express-rate-limit, multer, path-to-regexp) with no breaking changes. docker builder prune -a + image prune --filter until=24h reclaimed 278GB of system disk (/ went from 86% to 22%). All four project .env files chmod-ed to 600 to remove world-readable bits. Three zombie RemoteAGT containers cleaned up and nginx/conf.d/remoteagt.conf upstream pointed back at the DB-registered container.

  • Docs
    New documentation

    New page: AI Video Generation (Wan 2.2) — three-model comparison, parameter reference, distributed-GPU architecture diagram, measured performance, FAQ, security notes (KR + EN). Sidebar reorganized: old "Observability" group split into a new "🌱 AI & Generative" group with both the troubleshooting and the video-gen guides. Added distributed-GPU routing section to the AI Analyzer guide. Overview / Features / Index / Comparison / Pricing pages refreshed.

v2.1 April 11, 2026

🌱 Local Gemma 4 (Ollama) + Recovery Path Hardening

  • Feature
    Local Google Gemma 4 E4B Integration (Ollama-based, no API key required)

    Google DeepMind Gemma 4 E4B (Q4_K_M, 8B parameters, 9.6GB) is now served locally via an Ollama daemon — zero external API cost, no internet required, data never leaves the server. Model data lives on a separate data drive (OLLAMA_MODELS env), and the Ollama daemon is persisted as a user systemd unit (~/.config/systemd/user/ollama.service) with auto-restart on failure. NVIDIA CUDA 12 + dual GTX 1080 (16GB VRAM) detected automatically; ~46 tokens/s inference speed. Joins Anthropic Claude / Google Gemini as the third LLM engine — selectable per-project via "🌱 Gemma 4 E4B (Local · Free, Ollama)". All three service paths integrated: error analysis, AI chat (with full action-tag protocol), and auto code patch generation (JSON mode). Future variants (E2B / 26B / 31B) can be enabled by simply pulling the model and adding the dropdown option — the routing table is already wired. Configurable via OLLAMA_HOST environment variable for remote GPU servers.

  • Fix
    Container Recovery Logic — Eliminated Ghost Restart (Port Conflict Resolved)

    Blue-Green containers use the orbitron-{subdomain}-{deployHash} naming scheme, but the recovery logic was matching exact name orbitron-{subdomain} only — failing to find live containers and trying to spawn new ones, resulting in host port conflicts (Bind for 0.0.0.0:3480 failed). Now uses docker ps --filter "name=^orbitron-{sub}" prefix matching to prefer live containers. Exact match wins, otherwise the most recent deployHash variant is reused. When a live container is found, only the --restart unless-stopped policy is refreshed and container_id is synced — zero unnecessary container recreations.

  • Critical Fix
    True Root Cause of spawn E2BIG Identified and Fixed

    The recovery path was passing project.env_vars (an encrypted hex string) directly to dockerService.startContainer() without decryption. Object.keys(envVars) then iterated over the string's character indices — producing one -e N='X' flag per character (12,748 of them) and inflating the single command string to ~128KB, hitting MAX_ARG_STRLEN in execve. Fixed by adding a decryptProjectEnvVars() helper to server.js and applying it in the recovery loop before startContainer. Defense-in-depth: a type guard was also added to docker.js startContainer() that coerces non-object env_vars to {} with a warning.

  • Cleanup
    Broken env_vars Rows Recovered — Decryption Failing Log Spam Eliminated

    Four projects (Twinverse, RemoteAGT, KcontentsHub, K-ContentHub_DB) had env_vars encrypted with a previous JWT_SECRET, causing Decryption failing errors on every request and bloating error.log to 31MB. Safe recovery procedure performed: raw values backed up to backups/broken-env-vars-2026-04-11.json with full metadata, then DB rows reset to empty {} (re-encrypted with the current key). Original values can be recovered from the backup if the previous key is found later. Final reload monitoring confirmed zero Decryption failing / spawn E2BIG / Bind for 0.0.0.0 errors over a 30-second window.

  • UI
    AI Model Selector Updated

    The project settings AI model dropdown now exposes "🌱 Gemma 4 E4B (Local · Free, Ollama)". Selecting Gemma automatically hides the API key input fields and shows a green info banner explaining that the model runs locally with no external API or cost. The toggleAiKeyFields() JS function now recognizes the gemma- prefix.

v1.8.1 April 3, 2026

🔬 Code Review Critical Bug Hotfix

  • Hotfix
    Cloudflare Named Tunnel Creation Failure Fixed

    After _writeConfig was converted to async, it was called without await, causing configPath to be the string [object Promise]. This critical bug caused all Named tunnels to fail on startup.

  • Hotfix
    Duplicate stopTunnel Method Merged (Systemd Tunnels Never Stopped)

    stopTunnel() was defined twice — the second silently overwrote the first, making Systemd tunnel stop code dead. Now unified to properly handle Systemd, Quick, and Legacy tunnels.

  • Bug Fix
    Env Var Encryption Quote Bug, Health Check Port Mismatch & More

    Fixed: encrypted env vars wrapped in extraneous quotes causing decryption failure; health check using original port instead of auto-resolved port; 60-min timer not cleared on success; apk.txt newline regex matching literal \n instead of actual newlines.

  • Hotfix

v1.8.0 April 3, 2026

🛡️ Enterprise-Grade Stability & Security Optimization

  • New
    Docker Layer Caching Enabled by Default

    The --no-cache option that forced full rebuilds on every deployment has been removed. Docker build layer caching is now active by default, improving build speed by 2-3x when package.json hasn't changed. Use the DOCKER_NO_CACHE=true environment variable to disable caching when needed.

  • New
    60-Minute Deployment Timeout Protection

    All deployments now have an automatic timeout. Even if a process stalls due to network failures or infinite loop builds, it will auto-terminate after 60 minutes to prevent server resource exhaustion.

  • New
    Customizable Tunnel Domain via Environment Variable

    The default domain (twinverse.org) can now be changed via the TUNNEL_DOMAIN environment variable. If you own a Cloudflare domain, all project subdomains can be deployed to your custom domain.

  • Improved
    JWT Secret Security Hardening

    The hard-coded JWT fallback secret ('orbitron-secret-key') has been completely removed. The JWT_SECRET environment variable is now mandatory — the server will not start without it.

  • Improved
    DB Connection Pool Limits & Error Handling

    PostgreSQL connection pool now enforces max 20 connections, 30s idle timeout, and 5s connection timeout. Unexpected pool errors are automatically logged.

  • Improved
    512KB Log Limit & Auto Docker Image Cleanup

    Deployment logs exceeding 512KB are automatically truncated to prevent DB bloat. Docker images unused for 24+ hours are automatically pruned after deployment to reclaim disk space.

  • Improved
    Tunnel Exponential Backoff & Graceful Shutdown

    Cloudflare tunnel reconnection now uses exponential backoff (5s→10s→20s→max 5min). Graceful shutdown safely closes DB connection pools on server termination.

  • Improved
    Async I/O Conversion & Exec Buffer Optimization

    Nginx and tunnel config file writes have been converted to async operations, eliminating event loop blocking. exec() maxBuffer has been optimized from 50MB to 10MB for improved memory efficiency.

v1.7.1 March 24, 2026

🐳 Docker Dependency Auto-Injection & Local DB Port Forwarding Support

  • Feature
    Next.js System OS Dependency Auto-Injection (apk add)

    Analyzes package.json and automatically injects required OS system packages (ffmpeg, python3, cairo, etc.) into the Alpine Linux-based Docker image at build time if media-related packages (yt-dlp, ffmpeg) or image processing libraries (canvas) are present.   • Improvement
    Database External Port Auto-Mapping

    When deploying managed databases such as PostgreSQL or Redis, the assigned port is now automatically port-forwarded (-p) and connected to the internal Docker port (5432, 6379). This maintains internal network isolation while allowing developers to connect directly from the outside during local development via DB tools like Prisma Studio or DBeaver.
