# Orbitron 핵심 개념 (Core Concepts)

Orbitron을 200% 활용하기 위해 알아두면 좋은 3가지 핵심 개념과 작동 원리를 가볍게 설명합니다.

---

## 1. 깃허브 동기화와 무중단 배포 (Zero-Downtime CI/CD)

클라우드 서버에 접속(SSH)해서 매번 `git pull`을 치고 `pm2 restart`를 입력하던 시절은 끝났습니다. 
Orbitron은 당신의 GitHub 저장소와 완벽하게 동기화(Sync)됩니다.

### 작동 방식
*   당신이 로컬(VS Code 등)에서 코드를 짜고 GitHub의 `main` 브랜치에 코드를 푸시(Push) 하는 순간!
*   Orbitron의 웹훅(Webhook) 리스너가 이를 0.1초 만에 감지합니다.
*   클라우드에서 당신의 새로운 코드를 가져와 빈 컨테이너에서 새롭게 **빌드(Build)**를 시작합니다.

> ⚡ **완벽한 무중단(Zero-Downtime) 스위칭**
> 서버가 새로 업데이트 되는 (빌드하는) 도중에도 기존 사이트 접속자는 전혀 끊김이나 오류를 겪지 않습니다. 뒷단에서 새로운 컨테이너가 100% 정상 가동을 시작했다고 판별된 그 찰나의 순간에만, 사용자 트래픽의 물길(스냅 스위칭)을 돌려주기 때문입니다. 이전 버전은 그제야 조용히 소멸합니다.

---

## 2. 프로젝트 통합 환경 (Environments)

Orbitron에서는 **하나의 동일한 프로젝트 카드** 안에 여러 가지 생태계가 함께 존재할 수 있습니다. 
프론트엔드와 백엔드를 억지로 다른 프로젝트로 쪼갤 필요가 없습니다.

*   **Monorepo 지원:** 만약 여러분의 폴더 구조가 `frontend/` 프레임워크와 `backend/` 서버로 명확히 나뉘어 있다면? Orbitron 엔진이 이를 '풀스택 앱'으로 자동 감지하여, 프론트엔드는 정적 CDN으로, 백엔드는 Node.js 컨테이너로 알아서 두 번 배포해 줍니다!
*   **🧠 Smart Project Analyzer (2026.03 신규):** 단순 풀스택을 넘어서, 레포 안에 3개, 5개, 10개 앱이 있어도 전부 자동 감지합니다! 백엔드는 Docker로, 프론트엔드는 Cloudflare Pages CDN으로, 워커는 포트 없는 백그라운드 컨테이너로 — 각 서비스에 딱 맞는 타겟을 자동 선택합니다. [`orbitron.yaml`로 멀티 서비스 정의하기 →](#/iac)
*   **환경 변수 (Environment Variables):** 데이터베이스 비밀번호, API 키 등은 절대 코드에 하드코딩하면 안 됩니다. 프로젝트의 `Settings` 탭에서 환경 변수를 등록하면, 빌드와 런타임에 안전하게 해당 변수들이 시스템 안에 주입됩니다.

## 3. 격리된 컨테이너 (Isolated Containers)와 보안망

보안과 안정성은 Orbitron의 최우선 가치입니다.

### 컨테이너 오케스트레이션
모든 사용자 웹 서비스, 백그라운드 워커, 데이터베이스는 리눅스 기반의 완전히 독립된 **Docker 컨테이너** 공간 안에서 실행됩니다. A 사용자의 서버가 폭주하더라도 B 사용자의 서버는 1%의 성능 저하도 겪지 않도록 자원이 철저히 격리됩니다.

### 글로벌 터널 엣지 (Cloudflare Tunnels)
Orbitron에 배포된 프로젝트는 기본적으로 **"인터넷(외부 세계)에서 직접 IP를 치고 들어올 수 없는"** 상태로 시작됩니다. 
*   대신, 글로벌 1위 CDN 망인 Cloudflare 네트워크로 뻗어나가는 튼튼한 '비밀 터널'만을 단 하나 뚫어 놓습니다.
*   전 세계의 유저들은 해커들의 DDoS 공격을 원천 차단하는 Cloudflare Edge 망을 뚫고 들어와야만, 비로소 이 터널을 거쳐 여러분의 안전한 컨테이너에 도달할 수 있습니다.
*   이 과정에서 **HTTPS 무료 SSL 커스텀 도메인 (`*.twinverse.org`)**도 자동으로 적용됩니다.

## 4. 엔터프라이즈급 안정성 엔진 (2026.04 신규)

> ✨ **2026.04 업데이트**: Orbitron 코어 엔진에 대규모 안정성 및 보안 최적화가 적용되었습니다.

### 보안 강화
*   **JWT 시크릿 필수화**: 서버 시작 시 `JWT_SECRET` 환경변수가 반드시 설정되어야 합니다. 하드코딩된 기본값은 완전히 제거되었으며, 미설정 시 서버가 시작되지 않습니다.
*   **AES-256 암호화**: 모든 환경변수와 API 키는 AES-256-GCM 암호화를 거쳐 데이터베이스에 저장됩니다.

### 배포 안정성
*   **Docker 레이어 캐싱**: 기본적으로 Docker 빌드 캐시가 활성화되어 배포 속도가 **2~3배 향상**됩니다. 필요 시 `DOCKER_NO_CACHE=true` 환경변수로 캐시를 비활성화할 수 있습니다.
*   **60분 배포 타임아웃**: 배포가 60분을 초과하면 자동으로 중단되어 멈춘 프로세스로 인한 리소스 고갈을 방지합니다.
*   **로그 크기 제한 (512KB)**: 배포 로그가 512KB를 초과하면 자동 잘림 처리되어 데이터베이스 팽창을 방지합니다.

### 인프라 최적화
*   **DB 커넥션 풀**: PostgreSQL 커넥션 수 최대 20개, 유휴 30초, 연결 타임아웃 5초로 제한되어 커넥션 고갈을 방지합니다.
*   **Docker 이미지 자동 정리**: 배포 완료 후 24시간 이상 된 미사용 이미지를 자동 삭제하여 디스크 공간을 확보합니다.
*   **Graceful Shutdown**: SIGTERM/SIGINT 시그널 수신 시 DB 커넥션 풀을 안전하게 종료하고 10초 이내에 클린하게 프로세스를 마무리합니다.
*   **터널 지수 백오프**: Cloudflare 터널 연결 실패 시 5초→10초→20초→...최대 5분까지 점진적으로 재시도 간격을 늘려 네트워크 장애 시 과도한 요청을 방지합니다.
